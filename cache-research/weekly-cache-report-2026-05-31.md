## A. Production measurements and empirical studies

### 1. Databricks — "Accelerating LLM Inference with Prompt Caching for Open-Source Models" (May 22, 2026)

- Reference: [Databricks Engineering Blog — Accelerating LLM Inference with Prompt Caching for Open-Source Models on Databricks (May 22, 2026)](https://www.databricks.com/blog/accelerating-llm-inference-prompt-caching-open-source-models-databricks). Companion: prior-window [Databricks — How Superhuman and Databricks built a 200K QPS inference platform (May 8, 2026)](https://www.databricks.com/blog/how-superhuman-and-databricks-built-200k-qps-inference-platform-together).
- Summary: Prompt caching is now enabled by default for open-source models across batch, pay-per-token, and provisioned workloads on Databricks Model Serving — no user setup required. Caches are tenant-isolated, held in volatile memory only, and never persisted to durable storage; the post is explicit that this is a serving-layer optimization, not a logging surface. Production numbers from a GPT-OSS deployment: **2.5× throughput increase and 3× P50 latency reduction** vs the un-cached baseline at comparable cost.
- Novelty: Medium as a mechanism (it is prefix caching in production); high as a measurement. This is the second large-vendor production post in a month (after Superhuman + Databricks) that pins KV/prompt-cache wins to specific tenant-isolated, default-on numbers on open-source models — exactly the framing that lets enterprise operators justify the storage tier.
- Impact: High. With Anthropic, OpenAI, Google, AWS Bedrock, and now Databricks all defaulting prompt caching on, the operator default is shifting from "opt-in cost optimization" to "the platform handles this for you." The volatile-memory / tenant-isolation framing also nails down the data-residency posture that has blocked prompt caching at several regulated customers.
- Runtime evaluation: Yes, production — but partial. P50 latency and aggregate throughput multipliers on GPT-OSS; no P95/P99 tails, no cache-hit-ratio breakdown by workload type, and no head-to-head against the closed-model APIs operators are migrating from. The hit-ratio distribution would be the next number worth asking for.

### 2. NVIDIA — "Dynamo Snapshot: Fast Startup for Inference Workloads on Kubernetes" (May 27, 2026)

- Reference: [NVIDIA Developer Blog — NVIDIA Dynamo Snapshot: Fast Startup for Inference Workloads on Kubernetes (May 27, 2026)](https://developer.nvidia.com/blog/nvidia-dynamo-snapshot-fast-startup-for-inference-workloads-on-kubernetes/).
- Summary: A checkpoint-and-restore system for single-GPU inference workloads on K8s, built on Linux CRIU plus NVIDIA's cuda-checkpoint. The cache-relevant pieces: **KV-cache unmap before checkpoint** (so the snapshot does not pay to serialize attention state that will be regenerated anyway), parallel memfd restore, Linux native async I/O for CRIU memory restore, and a **GPU Memory Service (GMS)** that decouples large model weights from process state so many restoring containers can concurrently mmap the weights over GPUDirect Storage instead of streaming them serially.
- Novelty: Medium-high. Decoupling model weights from per-process snapshots and the explicit "unmap KV before checkpoint" pattern are the two original moves; the underlying CRIU/cuda-checkpoint pairing has been demoed before but not as a productized K8s primitive. The "model weights are a shared mmap-able tier, not part of the process image" framing is a clean storage-systems thought.
- Impact: High for inference fleet operators. Cold-start time on autoscaled inference pods is one of the few remaining places where caching can still pay 10–100× back, and Dynamo Snapshot is a credible attempt to make K8s replicas behave like CRIU-style restored processes rather than fresh starts.
- Runtime evaluation: Partial. Mechanism is well described and the optimizations are individually motivated, but the public blog text does not appear to attach a single end-to-end "X→Y seconds cold-start, on model Z" benchmark in the search-snippet view. Flagged as a gap — the numbers presumably exist in the GTC talk this post is summarizing.

### 3. vLLM × Novita AI — "PegaFlow for Production-Grade External KV Cache" (May 18, 2026)

- Reference: [vLLM Blog — PegaFlow: Production-Grade External KV Cache for vLLM (May 18, 2026)](https://vllm.ai/blog/2026-05-18-pegaflow). Companion: [PegaFlow source (github.com/novitalabs/pegaflow)](https://github.com/novitalabs/pegaflow).
- Summary: PegaFlow is a standalone Rust process that implements vLLM's external-KV connector contract. It offloads KV cache from GPU HBM into host memory and SSD on each vLLM node and **shares cache across nodes via RDMA**, so a single logical KV-cache pool spans the serving fleet and survives restarts and rolling redeploys. Architecturally it sits in the same niche as LMCache and Mooncake, but is the first major external-KV backend to ship as a separate Rust process rather than a Python in-process library.
- Novelty: Medium-high. The architectural choice (out-of-process Rust daemon, connector-interface-only coupling to vLLM) is the freshest part — it converts the external KV cache from a Python-extension dependency into a service-style component that can be operated and upgraded independently of the inference engine.
- Impact: Medium-high. PegaFlow is the third external-KV connector to land upstream in vLLM's ecosystem in the last two months (after LMCache and Mooncake-Store), and the operator-deployable Rust-process framing is what production teams have been asking for. The fact that it pairs with the v0.21.0 HMA/Mooncake consolidation from the previous week makes this the moment the connector ecosystem becomes plural.
- Runtime evaluation: No numbers surfaced in the snippet view of the blog. Flagged as a gap — the announcement appears to be a launch post rather than a benchmark study, and the operator question of "PegaFlow vs LMCache vs Mooncake at the same hardware and workload" is wide open.

### 6. Snowflake — "ZoRRo / Arctic RL: Enterprise RL Training, Breaking Performance Bottlenecks" (≈May 27, 2026)

- Reference: [Snowflake Engineering Blog — Zorro Enterprise RL Training (≈May 27, 2026)](https://www.snowflake.com/en/blog/engineering/zorro-enterprise-rl-training/). *Date is approximate — derived from search-snippet relative timing; confirm against the page itself before quoting.*
- Summary: Snowflake's RL stack ZoRRo (Zero Redundancy Rollouts), open-sourced as Arctic RL. The cache-relevant pieces are three: (a) **prefix-cache deduplication during rollout batching** so the shared prompt KV cache is computed once and reused across rollouts within a batch; (b) **Cascade Attention** that keeps the shared prompt KV cache resident in on-chip SRAM across requests rather than re-streaming it from HBM; (c) **Forest Cascade Attention** that groups requests by shared KV prefix and issues a grouped attention call per group. Headline: up to 3.5× faster RL iteration end to end.
- Novelty: Medium-high. Prefix-cache reuse in *RL rollout* (not serving) is the freshest of the three — it argues that the RL training inner loop should be treated as a prefix-cache-dominated workload and instrumented accordingly. The "forest" framing (group-by-prefix, grouped attention call) is a clean specialization of Cascade Attention for the rollout setting.
- Impact: Medium-high. Post-training is now a meaningful fraction of frontier compute, and RL rollout is structurally a prefix-cache problem (one prompt, many sampled continuations); making this explicit at the system level is a useful framing for everyone doing in-house RLHF/RLAIF. The open-source release (Arctic RL) is what makes this actionable outside Snowflake.
- Runtime evaluation: Yes, in training. 3.5× iteration speedup is the headline; per-component breakdown (how much of that 3.5× comes from Cascade vs Forest Cascade vs dedup) was not visible in the snippet view.

## B. Academic / idea-forward work

### 8. OSCAR — Offline Spectral Covariance-Aware Rotation for 2-bit KV Cache Quantization ([arXiv:2605.17757](https://hf.co/papers/2605.17757); v1 May 18, in window)

- Reference: [OSCAR (arXiv:2605.17757)](https://hf.co/papers/2605.17757).
- Summary: A genuinely deployable INT2 KV-cache. Existing simple rotations (Hadamard) fall over at INT2 because they are not aligned to what attention actually consumes. OSCAR estimates **attention-aware covariance structures offline** and derives fixed rotations plus clipping thresholds aligned to that covariance, then ships a **custom INT2 attention kernel compatible with paged KV-cache layouts** in SGLang / vLLM. Theoretical justification is paired with the integration, not just the algorithm.
- Novelty: High. The first credible INT2 KV-cache that survives realistic long-context reasoning workloads; attention-aware covariance is a genuinely new alignment criterion for the rotation step. This is the OSCAR-vs-TurboQuant question the field has been waiting for.
- Impact: High. If INT2 KV cache is genuinely deployable in long-context reasoning, the operator default for memory-bound workloads moves another notch lower; pairs with the prior-week vLLM TurboQuant reality-check (FP8 best for general use, TurboQuant only under memory pressure) by raising the "under memory pressure" upper bound.
- Runtime evaluation: Yes — strong, accuracy + system. Tested on Qwen3-4B-Thinking-2507, Qwen3-8B, Qwen3-32B, GLM-4.7 (358B); BF16 accuracy gap of 3.78 / 1.42 points on the smaller models, on-par at 32B / 358B; long-context RULER-NIAH out to 128K. **~8× KV memory reduction, up to 7× throughput at large batch, 3× bs=1 decode speedup vs BF16.**

### 9. KVServe — Service-Aware KV Cache Compression for Communication-Efficient Disaggregated LLM Serving ([arXiv:2605.13734](https://hf.co/papers/2605.13734); v1 ~May 13, boundary)

- Reference: [KVServe (arXiv:2605.13734)](https://hf.co/papers/2605.13734).
- Summary: Treats KV-compression as a **service-aware, adaptive runtime knob** in PD-separated and KV-disaggregated deployments rather than a static config. Three components: (1) a unified modular KV-compression strategy space supporting cross-method recomposition; (2) a **Bayesian Profiling Engine** that compresses offline search ~50× by producing a 3-D Pareto candidate set; (3) a **Service-Aware Online Controller** that combines an analytical latency model with a lightweight bandit to pick compression profiles under live SLO / bandwidth / workload-mix conditions.
- Novelty: High. First paper to formally cast KV compression as an online service-aware control problem; bandit-driven recomposition is novel and exactly the right framing for a disaggregated serving stack with mixed workloads on different links. Borderline-date (v1 May 13, just before the window) but absent from the prior report, so included as in-scope-new.
- Impact: High for the distributed-cache narrative. PD-separated and KV-disaggregated deployments are the architectural commitment vLLM, SGLang, and Mooncake are converging on; a principled online controller for compression along the K/V transfer link is the missing layer.
- Runtime evaluation: Yes. Integrated into vLLM, evaluated across datasets / models / GPUs / networks. **Up to 9.13× JCT speedup in PD-separated; up to 32.8× TTFT reduction in KV-disaggregated.**

### 10. RTPurbo — "Full Attention Strikes Back": Transferring Full Attention into Sparse within Hundred Training Steps ([arXiv:2605.16928](https://hf.co/papers/2605.16928); v1 May 16, boundary)

- Reference: [RTPurbo / Full Attention Strikes Back (arXiv:2605.16928)](https://hf.co/papers/2605.16928).
- Summary: Argues full-attention LLMs are already intrinsically sparse and only need ~hundreds of adaptation steps to be converted to a sparse-inference regime — pushing back on the native-sparse-pretraining (NSA-style) orthodoxy. Three observations: (i) only a small subset of attention heads requires full long-context processing; (ii) long-range retrieval lives in a low-dimensional subspace tractable by a **16-dim indexer**; (iii) useful token budget is query-dependent, so dynamic top-p beats fixed top-k. Realized as RTPurbo — keeps full KV cache only for "retrieval heads" and uses a lightweight token indexer elsewhere.
- Novelty: High. The dual claim that strong sparse inference is reachable without expensive native-sparse pretraining and that a 16-dim indexer suffices for long-range retrieval is mechanically and economically provocative; was the most-upvoted paper on HF in the window (~93 upvotes).
- Impact: Medium-high. If reproducible, the result reframes the sparse-attention investment curve — operators who have full-attention checkpoints can adapt in hundreds of steps rather than budgeting an NSA-style retrain.
- Runtime evaluation: Yes. Long-context benchmarks plus reasoning tasks; near-lossless accuracy preserved. **Up to 9.36× prefill speedup at 1M context, ~2.01× decode speedup.**

### 11. TIDE — Efficient and Lossless MoE Diffusion LLM Inference with I/O-aware Expert Offload ([arXiv:2605.20179](https://hf.co/papers/2605.20179); v1 May 19, in window)

- Reference: [TIDE (arXiv:2605.20179)](https://hf.co/papers/2605.20179).
- Summary: Targets **diffusion LLMs with MoE backbones** (e.g., LLaDA2.0). Observes temporal stability of expert activations *within* a diffusion block. Introduces **interval-based expert refresh** that updates expert placement I/O-aware, formulating the scheduling as a mathematical program that solves for an interval minimizing total I/O traffic plus CPU compute.
- Novelty: Medium-high. First diffusion-aware MoE expert-cache scheduling; bridges a previously underexplored intersection (dLLM × MoE × offload). The "temporal stability within a diffusion block" insight is the right one to exploit.
- Impact: Medium. Diffusion LLMs are still a research-stage workload, but the schedule-as-math-program framing transfers to autoregressive MoE expert offload as well.
- Runtime evaluation: Yes. Single-GPU + CPU system; **1.4× (LLaDA2.0-mini) and 1.5× (LLaDA2.0-flash) throughput** vs prior MoE-offloading baselines; lossless.

### 12. Quantized Keys Steal Attention — Bias Correction for KV-Cache Compression in Video Diffusion ([arXiv:2605.26266](https://hf.co/papers/2605.26266); v1 May 25, in window)

- Reference: [Quantized Keys Steal Attention (arXiv:2605.26266)](https://hf.co/papers/2605.26266).
- Summary: Identifies a **Jensen-bias** in quantized AR video diffusion: by convexity of `exp()` in softmax, quantization noise on cached keys **inflates cached-key contribution** at the expense of unquantized current-chunk keys — quantized cache effectively "steals" attention mass. Derives a closed-form per-attention-score correction (second-order Taylor, computed from quantization step sizes and query norm) with negligible compute and zero extra memory.
- Novelty: High conceptually. The Jensen-bias formalization is mechanistically new and applies beyond video diffusion — any setting that mixes quantized cached KV with unquantized current-token KV has the same convexity-of-softmax bias. The closed-form correction is elegant.
- Impact: Medium-high. Direct fix for INT2 video-KV; provides a generalizable correction recipe for the broader low-bit-KV cluster (OSCAR, TurboQuant, FibQuant) where this same bias is plausibly present and uncorrected.
- Runtime evaluation: Partial. Tested on MAGI-1, SkyReels-V2, HY-WorldPlay at INT2; recovers most of the BF16-quality loss and can outperform INT4 with 50% less memory. No detailed throughput/runtime study — accuracy-focused.

## Cross-cutting observations (this two-week window)

- **Production prompt-caching has gone default-on.** Databricks (entry #1) is now the fifth major serving platform — after Anthropic, OpenAI, Google, AWS Bedrock — to ship prompt caching as a no-setup-required default with explicit volatile-memory / tenant-isolation guarantees. The operator default has finished shifting from "opt-in cost optimization" to "the platform handles this for you"; the next question is hit-ratio reporting and contractual cache-isolation SLAs, not whether to enable it.
- **The external-KV connector ecosystem becomes plural.** PegaFlow (entry #3, out-of-process Rust daemon) joins LMCache and Mooncake as upstream-vLLM-compatible external KV cache backends. The architectural choice — operator-deployable service, not Python in-process library — is the new template. Pairs with the LMCache × Mooncake L2 adapter (entry #4 sub-item) where a single Mooncake pool can now back many LMCache MP processes: the connector contracts are starting to compose.
- **Cold-start cache management surfaces as its own layer.** NVIDIA Dynamo Snapshot (entry #2) productizes CRIU + cuda-checkpoint with explicit KV-cache-unmap and a GPU Memory Service that decouples model weights from process state. The framing — model weights are a shared mmap-able tier, not part of the process image — is the right storage-systems thought for autoscaled inference fleets and complements the SSD/DRAM/HBM tier consolidation from the prior fortnight.
- **Low-bit KV quantization splits along covariance lines.** OSCAR (entry #8) puts INT2 on the production-deployable map by aligning rotations to attention-aware covariance and shipping a paged-KV-compatible kernel; SAW-INT4 (additional context) argues INT4 is a "simple" regime where token-wise + block-diagonal Hadamard suffices. Together they map the "INT4 is simple, INT2 needs covariance" landscape; the Jensen-bias paper (entry #12) provides a generalizable correction primitive for the same family. This is a much more useful taxonomy than last fortnight's TurboQuant-vs-FP8 cross-fire.
- **KV compression becomes online control, not static config.** KVServe (entry #9) is the first paper to formally cast KV compression along a disaggregated K/V transfer link as a service-aware, bandit-driven control problem, with a Bayesian profiler that compresses offline search ~50×. As PD-separated and KV-disaggregated deployments become the operator default (vLLM HMA, Mooncake, SGLang HiCache), this is the kind of controller the production stack will need.
- **RL post-training is finally framed as a prefix-cache problem.** Snowflake's ZoRRo (entry #6) is the first major open-source RL stack to elevate prefix-cache deduplication, Cascade Attention, and Forest Cascade Attention to first-class scheduler concerns. Given post-training compute now rivals pre-training at the frontier, "RL rollout is a prefix-cache workload" is exactly the right framing — and the Arctic RL open source makes it actionable outside Snowflake.
- **The empirical-hygiene theme from the prior fortnight persists.** The MLSys 2026 LMCache invited-talk observation (entry #7) — context truncation in production halves prefix-cache hit ratio — is the same genre of finding as the prior fortnight's `PYTHONHASHSEED` caveat and the load-regime crossover where HBM prefix cache wins. The field is now publishing the gotchas it learned in deployment, not just new mechanisms; expect more of this.
