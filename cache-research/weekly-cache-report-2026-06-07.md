Recap: [Modular — Three Trends from MLSys 2026](https://www.modular.com/blog/three-trends-from-mlsys-2026).
- Summary: The prior week's report covered MLSys 2026's accepted-paper list; the conference itself ran in window, producing the Best Paper, awards, and several net-new cache/KV papers absent from prior reports. **BLASST** (NVIDIA Best Paper) is training-free dynamic block-sparse attention driven by a scalar softmax threshold that skips attention blocks, online-softmax statistics, value-block loads, and the GEMM — 1.52× prefill at 71.9% sparsity and 1.48× decode at 73.2% sparsity, already in TensorRT-LLM. **LMCache** got the inference invited-talk slot with the first canonical write-up of its enterprise KV-cache layer extracting/sharing KV across vLLM/SGLang engines (up to 15× throughput on multi-round QA / document analysis). **GhostServe** protects the streaming KV cache via erasure-coded parity shards in host memory: 2.7× faster checkpointing, 2.1× faster recovery, 1.2× median latency improvement. **SkipKV** does sentence-level coarse-grained KV eviction + generation suppression for reasoning models (matches FullKV accuracy on AIME-24 with 6.7× lower KV memory on R1-Qwen-14B). - Novelty: High overall. BLASST's scalar-threshold skipping of value-block loads + GEMM is the most novel sparse-attention mechanism shipped in window; GhostServe's erasure-coded KV-shadow is a fresh angle on fault tolerance for stateful inference; SkipKV's sentence-level eviction is the first to target reasoning-model regressions specifically; LMCache's invited-talk paper is the canonical reference operators will cite when discussing cross-engine KV sharing.
- Impact: High and immediate. BLASST is already integrated into TRT-LLM, so it will hit production through that path; the LMCache talk consolidates a year of operator messaging into one reference; GhostServe and SkipKV both map onto problems vLLM/SGLang have right now.
- Runtime evaluation: Yes across the board — prefill/decode speedups (BLASST), throughput on multi-round QA (LMCache), checkpoint/recovery time and latency (GhostServe), KV-memory reduction at iso-accuracy on reasoning benchmarks (SkipKV).

### 2. NVIDIA Dynamo Snapshot — KV-cache unmap for K8s fast startup (May 27, 2026)

- Reference: [NVIDIA Developer Blog — NVIDIA Dynamo Snapshot: Fast Startup for Inference Workloads on Kubernetes (May 27, 2026)](https://developer.nvidia.com/blog/nvidia-dynamo-snapshot-fast-startup-for-inference-workloads-on-kubernetes/). Companion: [ai-dynamo/dynamo v1.2.0 (June 2)](https://github.com/ai-dynamo/dynamo/releases).
- Summary: CRIU + `cuda-checkpoint` based checkpoint/restore for single-GPU inference workers, with first-class **KV cache unmap** to drastically shrink checkpoint size, parallel `memfd` restore, Linux AIO, and a GPU Memory Service that decouples model weights for concurrent restore over GPUDirect Storage. Reports **up to 21× startup-time reduction on gpt-oss-120b**. Dynamo v1.2.0 on June 2 added resilient KV routing (branch-sharded indexer) and a GPU Memory Service inter-pod sidecar, completing the cold-start story.
- Novelty: Medium-high. CRIU-on-GPU and cuda-checkpoint are well-known building blocks; declaring KV cache as the right thing to *unmap* (vs. preserve) at checkpoint time, then re-materializing only weights and runtime state on restore, is a clean operational insight. It is the K8s counterpart to LMCache's separation of KV state from model state.
- Impact: High. Inference autoscaling on K8s has been gated by minute-scale cold starts for 120B-class models; 21× brings the gpt-oss-120b cold-start curve into seconds-scale, which materially changes utilization economics.
- Runtime evaluation: Yes, on a real model (gpt-oss-120b) and a real substrate (K8s + GPUDirect Storage). End-to-end TTFT under autoscaled load is the natural next cut.

### 6. Apple — KV Prediction for improved TTFT (May 19, 2026)

- Reference: [Apple Machine Learning Research — KV Prediction for Improved Time to First Token (May 19, 2026)](https://machinelearning.apple.com/research/kv-prediction). Adjacent: [Apple — SpecMD: Speculative Expert Prefetching](https://machinelearning.apple.com/research/specmd-expert-prefetching) ([arXiv:2602.03921](https://arxiv.org/abs/2602.03921)).
- Summary: A small auxiliary model processes the prompt and **produces an approximation of the base model's KV cache**, which the base model then uses to start decoding earlier — reducing TTFT. Reports **15–50% relative accuracy improvement at fixed TTFT-FLOPs budgets on TriviaQA** and **up to 30% on HumanEval Python completion**. The accompanying SpecMD note attacks the MoE-expert-cache problem with a "Least-Stale" eviction policy reporting up to 85× fewer collision misses vs. LRU, >88% hit rate, and up to 34.7% TTFT reduction on OLMoE at a 5% (~0.6 GB) VRAM cache.
- Novelty: High. "Predict the KV cache instead of computing it" is a genuinely different angle from prefix-reuse and quantization, and is the most distinctive frontier-lab cache mechanism in window. SpecMD's Least-Stale exploits MoE access predictability specifically, a recurrent gap in expert-cache work.
- Impact: Medium-high for TTFT-sensitive serving (search, agents, on-device assistants). KV-prediction at fixed FLOPs is also a clean fit for Apple's on-device deployment shape.
- Runtime evaluation: Partial. Accuracy-at-fixed-FLOPs is measured cleanly; wall-clock TTFT on a serving stack and the prediction-model's training cost / online-update cost are still TBD. SpecMD reports hit rates and TTFT reduction at a small VRAM cache.

### 8. Snowflake — ZoRRo / Forest Cascade Attention for enterprise RL training (May 27, 2026)

- Reference: [Snowflake Engineering — Enterprise RL Training: Breaking Performance Bottlenecks (May 27, 2026)](https://www.snowflake.com/en/blog/engineering/zorro-enterprise-rl-training/).
- Summary: ZoRRo (Zero-Redundancy Rollouts) targets enterprise RL training; the cache-relevant piece is **Forest Cascade Attention**, an extension of cascade attention to multi-prefix RL workloads: requests sharing KV-cache prefixes are grouped at decode time and the shared prefix KV is reused from on-chip GPU SMEM across all requests in each group. Paired with split-attention for training and speculative decoding for generation.
- Novelty: Medium-high. Cascade attention is known; *forest* cascade — multiple shared prefixes co-resident in SMEM with per-prefix request groupings — is the natural RL-rollout generalization, and the right one for tree-shaped rollouts.
- Impact: Medium-high for RL training providers (Anyscale, OpenRLHF-style stacks) and for any serving workload with branched rollouts (search, agents with backtracking).
- Runtime evaluation: Partial — blog cites ZoRRo throughput gains, but the per-mechanism contribution of Forest Cascade vs. other ZoRRo pieces is not broken out separately.

## B. Academic / idea-forward work

### 12. VeriCache — turning lossy KV cache into lossless LLM inference ([arXiv:2605.17613](https://arxiv.org/abs/2605.17613); ~May 25, 2026)

- Reference: [arXiv:2605.17613](https://arxiv.org/abs/2605.17613). Authors include the LMCache/UChicago lineage (Yao, Shen, Du, Feng, Liu, Lu, Jiang).
- Summary: A speculative-style framework that **drafts tokens with a compressed KV cache and verifies them against the full KV cache**, guaranteeing identical output to full-KV decoding while keeping most of the throughput of lossy compressors. Reports up to **4× speedup vs. full-KV at parity**.
- Novelty: High. First framework providing exact-equivalence guarantees layered over arbitrary lossy KV compressors — orthogonal to compressor choice (quantization, eviction, low-rank, etc.) and complementary to speculative decoding on the model side.
- Impact: Potentially high. If the wrapper composes cheaply with existing compressors, it neutralizes the "but accuracy regressed" objection that has stalled production adoption of aggressive KV compression for a year.
- Runtime evaluation: Yes — throughput vs. full-KV decoding across multiple compression algorithms; deployment substrate not yet disclosed in the snippet, but the LMCache lineage suggests a vLLM/Mooncake integration is plausible.

### 13. ObjectCache — layerwise object-storage retrieval for KV reuse ([arXiv:2605.22850](https://arxiv.org/abs/2605.22850); ~May 21, 2026)

- Reference: [arXiv:2605.22850](https://arxiv.org/abs/2605.22850).
- Summary: Stores prefix KV cache on **S3-compatible object storage** instead of remote DRAM pools. Co-designs a hash-addressed chunk protocol with **layerwise transfer overlapped with per-layer GPU compute**; the storage server assembles layer-major payloads delivered via RDMA. For 64K context: **only 5.6% TTFT overhead vs. an optimized local layerwise baseline**; for 4K contexts: 56–75 ms cost.
- Novelty: High. The first system to put *runtime* KV cache reuse onto S3-shaped object storage with TTFT-competitive results — distinct from Google's Lustre/Rapid offering (which is filesystem-backed) and from MinIO MemKV (entry #5, RDMA-direct JBOF, no object semantics on the data path). Layer-major payload assembly inside the storage server is the right design call for this tier.
- Impact: Strong if the numbers hold. Object storage is the cheapest, most-deployed durable tier; if KV cache can ride on it at +5.6% TTFT at 64K, the economics of "always-cached" multi-day agent histories become trivial.
- Runtime evaluation: Yes — TTFT against a strong local-layerwise baseline at multiple context sizes; baseline vs. true full-prefill comparison and a multi-tenant hit-rate study are the natural follow-ups.

### 14. Lodestar — online-learning router for LLM inference clusters ([arXiv:2606.00946](https://arxiv.org/abs/2606.00946); May 31, 2026)

- Reference: [arXiv:2606.00946](https://arxiv.org/abs/2606.00946). Affiliation: UIUC, ByteDance, Univ. Edinburgh.
- Summary: A cloud-native router for vLLM clusters that learns an online reward predictor from per-request snapshots — instance state, request features, observed performance — and routes to maximize a reward (e.g., min TTFT). Adapts to changing workloads **within ~5 minutes**. Reports **1.41× average / 1.47× P99 TTFT improvement vs. a prefix-cache + load-aware baseline; up to 4.4× on heterogeneous clusters**.
- Novelty: Medium-high. KV-cache-aware routing exists; *online-learning* routing that simultaneously handles cache reuse, heterogeneous accelerators, and nonlinear latency is fresher and matches the vLLM v0.22 Session-Aware Agentic Routing direction (entry #3) on the research side.
- Impact: High for multi-tenant inference clusters with heterogeneous GPUs (the 4.4× heterogeneous-cluster number is the headline if it holds). Lodestar is the kind of system that ought to land in llm-d as a scheduler.
- Runtime evaluation: Yes — TTFT averages and P99 against a prefix-cache + load-aware baseline. Workload mix and scale of the heterogeneous-cluster experiment will determine how broadly the 4.4× generalizes.

### 15. AsymCache — kernel-aware KV-cache eviction tuned for TTFT/TPOT ([arXiv:2606.02964](https://arxiv.org/abs/2606.02964); June 1, 2026)

- Reference: [arXiv:2606.02964](https://arxiv.org/abs/2606.02964). Affiliation: Peking University.
- Summary: Jointly considers KV-cache hit rate and **position-aware recomputation cost**, plus an adaptive chunking scheduler **aligned with GPU attention-kernel performance**. Reports **1.90–2.03× TTFT speedup and 1.62–1.71× TPOT reduction** vs. prior baselines.
- Novelty: Medium-high. First eviction policy to incorporate kernel-level execution latency of attention as a cost term — most eviction work models hit-rate alone. The chunking-scheduler/kernel co-design is the clean systems contribution.
- Impact: Medium-high. If the kernel-aware term composes with PagedAttention and FlashAttention-4, this becomes the eviction baseline for the next generation of decoders.
- Runtime evaluation: Yes — TTFT and TPOT against latest baselines. Hardware and model coverage will determine reproducibility outside the paper.

### 16. KVarN — variance-normalized 2-bit KV quantization beating FP16 throughput in vLLM ([arXiv:2606.03458](https://arxiv.org/abs/2606.03458); June 2, 2026)

- Reference: [arXiv:2606.03458](https://arxiv.org/abs/2606.03458). Affiliation: Huawei.
- Summary: Calibration-free 2-bit KV quantizer combining **Hadamard rotation with dual-scaling variance normalization on K and V**, addressing autoregressive error accumulation specific to reasoning workloads. Reports SOTA on MATH500, AIME24, HumanEval at 2-bit, with **3–5× context capacity** and **higher-than-FP16 throughput in vLLM**, shipped as a one-flag vLLM backend.
- Novelty: Medium-high. Dual-axis variance normalization across K and V is the new piece on the algorithmic side; the calibration-free property and "ship as a vLLM flag" reduce adoption friction significantly.
- Impact: Strong if the FP16-throughput-beating claim survives independent measurement — that would be the first 2-bit KV result on the right side of the throughput Pareto rather than just the capacity Pareto.
- Runtime evaluation: Yes — vLLM throughput plus reasoning benchmarks. Pairs directly with last week's vLLM TurboQuant accuracy study (which concluded FP8 was still the right default) — KVarN's "we beat FP16 throughput" claim is exactly what TurboQuant did *not* deliver.

### 17. Vortex — programmable sparse-attention serving for AI agents ([arXiv:2606.06453](https://arxiv.org/abs/2606.06453); June 4, 2026)

- Reference: [arXiv:2606.06453](https://arxiv.org/abs/2606.06453). Affiliation: CMU, Rice, NUS (Infini-AI Lab).
- Summary: A Python-embedded frontend on a **page-centric tensor abstraction** for expressing sparse-attention algorithms, with a backend tightly integrated into modern LLM serving stacks. Demonstrates **AI agents generating, benchmarking, and refining sparse-attention algorithms**; best agent-designed variants reach **4.7× throughput over full attention with quality preserved**.
- Novelty: High. The first serving substrate explicitly designed for *agent-driven* sparse-attention algorithm design — and the meta-result (agents can find competitive sparsity patterns automatically) is interesting beyond the substrate.
- Impact: Medium-high. Whether this lands as a serving runtime or as a research playground depends on integration. The page-centric tensor abstraction itself looks reusable.
- Runtime evaluation: Yes — throughput vs. full attention on a serving stack with multiple agent-designed sparse-attention algorithms.

## Cross-cutting observations (this period)

- **MLSys 2026 happened in window, and its center of gravity is unambiguously KV-cache systems.** Modular's recap pegs 6 LLM-serving sessions (2× prior year), and the in-window oral cluster (BLASST, LMCache invited talk, GhostServe, SkipKV, FlexiCache, Kitty, MorphServe, TriInfer, HELIOS) collectively says: quantization × sparsity is now a joint optimization problem, hierarchical/tiered KV memory is the default, and PD-disaggregation is the TCO story. This is the conference confirming a year of vendor-blog claims.
- **The "G3.5" / CMX KV-cache tier is now a vendor product category.** Three storage vendors (MinIO MemKV, WEKA NeuralMesh, VAST Data) all landed first-class KV-cache-tier disclosures in the same fortnight, all targeting NVIDIA's CMX hierarchy and BlueField-4 DPUs. Pair with vLLM's HMA + Mooncake connector consolidation last week and NVIDIA Dynamo + NIXL this week and you have the engine ↔ transport ↔ storage triangle finally drawn. The Tensormesh Series A with strategic checks from AMD, NVIDIA, and CoreWeave (entry #7) is the same shape, viewed from the software side.
- **Cold-start as the new TTFT.** Two independent disclosures hit the same number from different angles — NVIDIA Dynamo Snapshot's **21×** startup-time reduction on gpt-oss-120b (entry #2) and Modal's **40×** cold-start reduction (50 s vs. 2000 s) via CRIU + cuda-checkpoint. The common pattern: unmap the KV cache at checkpoint, restore only the durable state, rebuild the rest. K8s autoscaling for 100B-class models is moving from minutes to seconds.
- **Lossy KV cache, but lossless inference.** VeriCache (entry #12) is the cleanest articulation yet of the pattern that has been emerging across the year: layer a speculative-style verification wrapper over any lossy compressor so that the *final output* is bit-identical to full-KV. If this composes cheaply with TurboQuant, KIVI, Kitty, etc., it removes the largest objection to aggressive KV compression in production.
- **Frontier labs converged on the same attention/cache recipe.** Anthropic, Cohere, Google (Gemma 4 + Gemini 3.5), and Microsoft (MAI-Thinking-1) all shipped in window. Across them, the pattern is striking: sliding-window local + global attention with KV sharing in global layers; MoE-aware speculative decoding; explicit API-level cache economics. This is the production default crystallizing across the field, not a single-lab pattern.
- **Cache evaluation methodology continues to be a contribution in its own right.** SGLang v0.5.12.post1's HiCache/HiSparse GSM8K 0.825 → 0.960 fix (entry #10) and the "Structural Protection Dominates Scoring" negative result (Additional Context) are the canonical examples this period: hierarchical KV stacks can quietly produce double-digit accuracy regressions, and the entire eviction literature can collapse under one missing structural-protection ablation. Last week's LMCache `PYTHONHASHSEED` caveat sits in the same tradition.
